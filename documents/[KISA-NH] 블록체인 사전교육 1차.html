해커톤에서 Dapp 개발 노하우
  • 아이디어 빌드
    • 신뢰받는 제 3자가 필요한 아이디어 인가?
      • 블록체인은 제 3의 중개기관을 배제할 수 있음
      • 공인인증서, CA 등..
    • 데이터를 나눠서 저장해야 하는가?
      • 기존 시스템 과의 데이터 저장 차이점을 인지할 것
        • 하나의 노드가 변화하면 수 많은 노드가 상태값을 변경하는 구조
        • 변경이 많이 일어나거나, 소량의 정보만을 저장하는 경우 블록체인이 적합
        • 절대 분할 저장이 아님, 각 노드에 데이터가 duplicate된 것
        • 대용량 데이터 저장 및 분산 데이터 저장이 필요 없을 경우 기존 구조를 유지할 것
    • 빠른 처리속도가 필요한가?
      • 낮은 확장성을 인지해 서비스 모델에 적용할 것
      • EC2에 비해 Ethereum의 속도는 100만배 차이난다.
      • 확장성
        • 처리속도, 검증속도, 전파속도, 데이터 크기, 노드 개수, 다양성, 거버넌스
    • 의사결정이 타인에 의해 이루어져도 되는가?
      • 서비스 모델의 내용 및 의사 결정 방식에 따라 블록체인 도입 여부 결정
        • 블록체인은 토큰이 발행되고, 투자하거나 컴퓨터 파워를 제공한 사람에게 의사결정 권한이 나눠짐
        • 발행 된 후 내 통제 권한은 떠난 것
        • 의사결정 과정이 투명하게 공개되어야 함.
        • 네트워크 참여자들에 의해 의사결정 방향이 왜곡될 수 있음
        • 단순 솔루션 혹은 개인적인 목적일 경우 블록체인을 사용하면 안됨
      • Steam Blockchain
        • 투표에 근거한 블록 결정
    • 거래의 투명성이 필요한 아이디어 인가?
      • 데이터의 투명성을 입증해야하는 서비스 모델의 경우 블록체인 적용이 타당
      • 모든 데이터가 암호화 되어 시간순으로 블록이 연결됨, 비밀이 없음
      • 온체인에 탑재가 한번 되어버리면 지울 수 없음
      • 선거와 같은 내용에 적용이 가능함
    • 이 다섯가지가 부합된다면 좋은 아이디어 일 수 있음

  • 데이터 공개 여부 결정
    • public / private 
      • 네트워크 참여자가 많으면 당연히 느려짐


개발 시작
  • 온체인 개발 요소를 구분해서 설계
    • 네트워크에 참여하는 모든 노드, 비용이 비쌈
    • Contract Business logic의 복잡성 최소화
      • 현재 파일 공유, 머신러닝 등의 복잡한 프로그램을 돌리는 것은 거의 불가능
      • 간단한 조건문, 인증 철자 등 단순 태스크에 적합
    • 꼭 필요했을때만 온체인 탑재하는 것이 맞음
    • 전체 일이 다 들어가면 기하급수적으로 느려짐
    • 사용하면 안되는 경우
      • 크고 복잡한 데이터, 개인정보, 기밀 데이터, 데이터 수집을 위해 외부 서비스를 이용해야 하는 경우
    • D-Live의 경우
      • 채팅은 Off-Chain, Vote(지불관련)은 On-chain으로 구성
  • 수수료를 줄이기
    • 코드 실행 횟수 줄이기
    • 스마트 컨트랙트가 특정 시점이 되면 종료하도록 하기 위해, 실행 코드당 가격을 매김
    • 루프 반복 횟수를 줄여야 함
    • 사용되는 가스 총량이 가스 한도 초과 시, 전부 원상태로 롤백됨
  • 한번 Deploy된 코드는 바꿀 수 없음
    • 비가역성
      • Code update시, Instance 재구성을 하거나 Proxy 역할을 하는 Contract를 써야 함
      • 한번 배포된 계약은 사업 주체의 통제에서 벗어남, 비즈니스 로직의 오작동을 방지하기 위한 대책 필요
    • 디버깅에 신경쓰기
      • 버전에 따라 코드의 내용이 상이함
      • 온라인에 충분한 자료가 없기 때문에..
      • solidity 자체 결함도 존재
    • Fail-safe Mode 매커니즘 - 상용 단계에서 진행
      • 오류 방지 매커니즘을 포함시켜, 향후 발생 가능한 심각한 해킹, 버그에 대비
      • 임의의 이더가 유출 되었나요? 토큰의 합계까 계약의 잔액과 같은가요?
      • 자가 폭파등
    • Smart Contract에 저장하는 Token의 양 제한하기
      • Source code, Compiler, Platform에 버그가 있을 경우 자금 손실 우려
  • 외부 데이터 호출에 유의하기
    • Oraclize 사용
      • smart contract에서 내부 데이터 / 타 smart contract로부터 받는 데이터 외에 외부데이터 사용시, HTTP call이 불가능
      • 이더리움 네트워크 외부와 정보를 연결해주는 서비스를 'oracle'이라고 함
      • 이더리움 네트워크와 외부의 정보를 연결해주는 서비스를 오라클이라고 함
        • 예를 들어 100개의 노드가 네이버 사이트를 요청할때, 노드 성능에 따라 리턴값이 다를 수 있음(시간에 따라) 이런 상황을 방지하기 위함
        • loop chain에서는 portal이라고 함
      • Oracllize가 단독 실행되어 해당 정보를 각 노드에게 나누어줌
  • 스마트 컨트랙트 취약점 주의
    • 코드 투명성으로 인한 취약점 노출
      • 누구나 컨트랙트 코드를 조회할 수 있음, 기술 취약점이 노출되기 쉬움
    • 재진입 공격
      • 외부 계약 호출의 실행이 완료되기도 전에 새로운 호출 허용시 발생
      • recursive call
    • 접근 제어자 & delegatedcall 유의
      • delegatecall : 권한 위임하여 실행시키는 것
        • 해당 호출 대상에 누수 없는지 꼭 확인할 것
      • fallback : 다른 함수에 매칭되지 않을때, 기본으로 호출되는 함수
        • 매칭되는 transaction이 없으면 해당 내용으로 강제 실행
  • Check-Effects-Interactions Pattern
    • 검사 우선의 패턴
      • 내부에 있는 것을 모두 처리하고 외부 처리하기
  • 기타 유의 특성
    • 랜덤 값 생성 불가
      • 컨트랙트 안에서 랜덤 값을 생성하거나, 실행하는 모델은 불가
      • 블록의 해시 혹은 transaction을 이용한 랜덤 값 이용은 가능
    • 부동 소수점 처리 불가
      • 모든 연산은 정수단위로(CPU에 따라 처리 방식이 다를 수 있기 때문)
    • 시간에 따라 행동하는, 실행 시간에 따라 내용이 바뀌는 모델 구현이 어려움
      • 블록의 시간 혹은 트랜잭션 시간으로 대체 가능
